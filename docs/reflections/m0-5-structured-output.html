DOOH Control Tower — Reflection M0.5

Structured output: typed tool results, schema for free

about() from M0.3 returns a string — the agent gets prose it has to re-parse. health_check() returns a Pydantic BaseModel — FastMCP auto-generates a JSON outputSchema from the type hint, the wire response includes a structuredContent field the agent can read by name, and downstream tools (mcp-ui, Pydantic AI) can validate against the same schema. This is the first chunk where the *shape of the data* is the lesson, not the transport.

Big picture

A typed return value flows through three FastMCP layers: Pydantic generates the JSON Schema from the type hints; FastMCP attaches that schema to the tool's tools/list response; and on tools/call, FastMCP runs the tool, validates the result against the schema, and emits both a human-readable content[text] rendering and the typed structuredContent object.

One type hint, three wire-level artifacts. The agent gets prose and structured data and a contract to validate them against — from a single line of code.

  4. Added tools/__init__.py as a side-effect import (each module registers its tools via @mcp.tool() at import time). server.py triggers the package import at the bottom of the file to avoid a circular import.

String vs typed return — what the agent sees

Both tools live in the same server, both ride the same transport. The only difference is the return type. Watch the wire diverge.

about returns: str
outputSchema (from tools/list)
{
  "properties": {
    "result": {"title": "Result", "type": "string"}
  },
  "required": ["result"],
  "title": "aboutOutput",
  "type": "object"
}
structuredContent (from tools/call)
{
  "result": "DOOH Control Tower — a chat-driven operational tool ..."
}

FastMCP wraps the string in {result: ...} so the wire shape is always an object. Agent still has to parse prose to get any specific fact out.

health_check returns: HealthCheck
outputSchema (from tools/list)
{
  "$defs": {
    "ComponentStatus": {
      "properties": {
        "name": {"type": "string"},
        "status": {"enum": ["ok","degraded","down"], "type": "string"},
        "detail": {"anyOf": [{"type":"string"},{"type":"null"}], "default": null}
      },
      "required": ["name","status"],
      "type": "object"
    }
  },
  "properties": {
    "status": {"enum": ["ok","degraded","down"], "type": "string"},
    "timestamp": {"format": "date-time", "type": "string"},
    "components": {"items": {"$ref": "#/$defs/ComponentStatus"}, "type": "array"}
  },
  "required": ["status","timestamp","components"],
  "type": "object"
}
structuredContent (from tools/call)
{
  "status": "ok",
  "timestamp": "2026-05-31T11:45:48.909921Z",
  "components": [
    {"name": "mcp", "status": "ok", "detail": null}
  ]
}

Agent can ask "what's the status of mcp?" and read components[0].status directly — no prose parsing.

How the schema is generated (Pydantic → JSON Schema)

FastMCP doesn't generate schemas itself — Pydantic does. FastMCP just collects them from the tool's return-type hint and attaches them to the protocol response. Every type hint maps to a JSON Schema construct:

Python type JSON Schema Why it matters to an agent
Literal["ok","degraded","down"] {enum: ["ok","degraded","down"]} Agent knows the finite set up front — can branch deterministically.
datetime {type: "string", format: "date-time"} Wire is RFC-3339 string. Agents and validators recognize the format.
str | None {anyOf: [{type:"string"},{type:"null"}], default: null} Optionality is explicit — agent doesn't guess whether a field can be missing.
list[ComponentStatus] {type: "array", items: {$ref: "#/$defs/ComponentStatus"}} Nested types are extracted once into $defs and referenced — no duplication, cycle-safe.
BaseModel subclass {type: "object", properties: {...}, required: [...]} Field names, types, and required-ness are all explicit.

A future M2 tool returning list[Campaign] with nested Targeting and Schedule models will get a multi-level $defs tree the same way — no extra work.

The code

src/dooh_control_tower/mcp/tools/health.py
from datetime import UTC, datetime
from typing import Literal

from pydantic import BaseModel

from dooh_control_tower.mcp.server import mcp


class ComponentStatus(BaseModel):
    name: str
    status: Literal["ok", "degraded", "down"]
    detail: str | None = None


class HealthCheck(BaseModel):
    status: Literal["ok", "degraded", "down"]
    timestamp: datetime
    components: list[ComponentStatus]


@mcp.tool()
def health_check() -> HealthCheck:
    """Report the operational health of ..."""
    return HealthCheck(
        status="ok",
        timestamp=datetime.now(UTC),
        components=[
            ComponentStatus(name="mcp", status="ok"),
        ],
    )

Why this way (decisions and trade-offs)

Why a structured return type at all? Why not just JSON-stringify in the tool body?

You could have the tool return a string of JSON. But then:

  • The agent gets a blob it has to JSON.parse with no schema to validate against.
  • The wire outputSchema is wrong — it'll say "string" when the content is structured.
  • Downstream consumers (mcp-ui surfaces in M2, Pydantic AI client in Phase 2) can't introspect the shape.
  • Refactoring the model is silent — no compile-time error if a field is renamed.

Pydantic return types make the shape part of the contract, not the convention.

Why drop uptime_seconds?

Uptime sounds simple but semantics shift by transport: stdio uptime is "since Claude Desktop spawned this subprocess" (resets on every chat). HTTP uptime is "since uvicorn worker booted" (different across workers). Per-tool-call uptime is meaningless. Multi-worker mode multiplies the ambiguity.

Until we have one canonical answer to "what is uptime of", reporting it would mislead. Dropped until we have it figured out.

Why list[ComponentStatus] instead of dict[str, ComponentStatus]?

A dict-keyed-by-name generates {additionalProperties: {$ref: ...}} — a "container of arbitrary string keys." Self-documenting for humans, less so for agents and validators: the schema doesn't tell them what keys are possible.

A list of typed objects with an explicit name field gives the schema an array of definite shape. Easier to iterate, easier to validate, easier to extend with order or grouping later.

Why move about into tools/ too?

If tools/ exists, having one tool live elsewhere creates a "why is this one different?" question every future reader has to answer. Consistency removes the cognitive lump for free.

It also makes server.py contain exactly what it's named for — the server instance + transport entry point — and nothing else.

Why side-effect imports and the bottom-of-file import in server.py?

Each tool module decorates its function with @mcp.tool() at module-load time. That's how FastMCP discovers tools — by being decorated, not by a registry call. So importing the module is the registration.

The catch: tool modules from ...server import mcp. If server.py tried to from .tools import ... at the top, Python would re-enter server.py before mcp was defined — circular ImportError.

Putting the import at the bottom (after mcp is defined) breaks the cycle. The noqa: E402, F401 tells ruff "this is intentional, not a violation." The alternative — extracting mcp into its own instance.py module — works too, and might be worth doing if the bottom-of-file pattern feels weird later.

Why add pydantic as a direct dependency?

Pydantic was already installed via mcp[cli]'s transitive requirements. But we now use Pydantic in our own code — it's a real direct dependency.

Pinning explicitly: (1) prevents a future MCP SDK update from silently downgrading or removing the dependency; (2) signals intent to future readers — "we depend on Pydantic on purpose, not by accident."

Workflow notes

No grill-with-docs this chunk — and why that was right
  • The skills habit memo: skills fire at high-leverage moments, not every chunk. M0.5 is domain-shaping (what does the tool return?), not library-binding (which MCP feature do we adopt?).
  • FastMCP's "return any Pydantic model and you get a schema" path is well-trodden — no surprising API knob to validate.
  • The decisions that needed airing (response shape, list vs dict, code location) were product/structure calls, not "does this API actually exist?" calls.
  • Risk avoided: reflexive grill-with-docs on every chunk would dilute the habit. Saving it for moments where docs change the answer keeps the signal strong.
