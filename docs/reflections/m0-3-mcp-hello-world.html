MCP (Model Context Protocol) is a JSON-RPC protocol that lets an LLM host — like Claude Desktop — discover and call tools that you expose from your own process. FastMCP is the Python SDK's high-level primitive: you decorate a typed function with @mcp.tool() and it becomes a callable tool with an auto-generated JSON schema. stdio is the simplest transport: the host launches your server as a child process and they exchange JSON-RPC frames over stdin/stdout.

How it works

The 24-line server

# stdio MCP wire uses stdout for JSON-RPC frames — never print() in this module.
from mcp.server.fastmcp import FastMCP

mcp = FastMCP(name="DOOH Control Tower")


@mcp.tool()
def about() -> str:
    """Describe the DOOH Control Tower MCP server: purpose, thesis, and current build phase."""
    return (
        "DOOH Control Tower — a chat-driven operational tool for Digital Out-of-Home "
        "advertising. Thesis: the agent decides WHAT to show, the app decides HOW to "
        "render. Phase 1 build: MCP + Claude Desktop + mcp-ui. The DOOH adtech "
        "(adserver, campaign manager, reporting) is the canvas; the chat-driven "
        "operational tool is the product."
    )


def main() -> None:
    mcp.run()


if __name__ == "__main__":
    main()

Every visible piece pulls weight: the comment defuses a future footgun, the decorator registers the tool and generates its JSON schema from the type hints, the docstring becomes the tool's description in that schema, and mcp.run() defaults to stdio when called without arguments.

The protocol handshake

sequenceDiagram participant H as MCP host
(Claude Desktop, Inspector, client SDK) participant S as dooh-mcp
(FastMCP) Note over H,S: 1. Host spawns server as a child process
and connects its stdin/stdout H->>S: initialize { protocolVersion, capabilities, clientInfo } S-->>H: result { protocolVersion, capabilities, serverInfo } H->>S: notifications/initialized Note over H,S: 2. Host asks what tools exist H->>S: tools/list S-->>H: result { tools: [{ name: "about", description, inputSchema }] } Note over H,S: 3. Host (or the LLM) decides to call one H->>S: tools/call { name: "about", arguments: {} } S-->>H: result { content: [{ type: "text", text: "DOOH Control Tower …" }] }

Three round-trips. The first is the protocol negotiation. The second is tool discovery — this is the only place the LLM "learns" what tools exist, which is why the docstring and type hints matter: they become the tool's specification. The third is the actual call.

FastMCP vs low-level Server — same tool, two altitudes

FastMCP is sugar over the lower-level mcp.server.Server primitive. To see what FastMCP is doing for us, here's the same about() tool implemented at both levels.

FastMCP (what we shipped)

from mcp.server.fastmcp import FastMCP

mcp = FastMCP(name="DOOH Control Tower")


@mcp.tool()
def about() -> str:
    """Describe the … server."""
    return "DOOH Control Tower — …"


def main() -> None:
    mcp.run()
  • ✓ Sync function, no asyncio in user code
  • ✓ JSON schema auto-derived from type hints + docstring
  • ✓ list_tools + call_tool routing automatic
  • ✓ stdio transport implicit in mcp.run()

Low-level Server (what FastMCP hides)

import asyncio
from mcp.server import Server, NotificationOptions
from mcp.server.models import InitializationOptions
from mcp.server.stdio import stdio_server
import mcp.types as types

server = Server("DOOH Control Tower")


@server.list_tools()
async def list_tools() -> list[types.Tool]:
    return [types.Tool(
        name="about",
        description="Describe the … server.",
        inputSchema={"type": "object",
                     "properties": {},
                     "required": []},
    )]


@server.call_tool()
async def call_tool(name, arguments):
    if name == "about":
        return [types.TextContent(
            type="text", text="DOOH Control Tower — …")]
    raise ValueError(f"unknown: {name}")


async def main():
    async with stdio_server() as (read, write):
        await server.run(read, write,
            InitializationOptions(
                server_name="DOOH Control Tower",
                server_version="0.1.0",
                capabilities=server.get_capabilities(
                    notification_options=NotificationOptions(),
                    experimental_capabilities={}),
            ))

asyncio.run(main())
  • ✗ Async by default — manual asyncio.run()
  • ✗ Hand-write the JSON schema in inputSchema
  • ✗ Manual routing in call_tool for each tool name
  • ✗ Manual stdio_server() + InitializationOptions boilerplate

Takeaway: FastMCP is roughly 14 lines of meat vs. 35 lines of plumbing, and the semantics are identical on the wire. For Phase 1, where every chunk adds one or two tools, FastMCP is pure win. The low-level Server only earns its keep when you need custom protocol behavior that the decorator API doesn't expose (e.g., streaming partial results, dynamic tool catalogs, progress notifications).

▸ Why this way (decisions & tradeoffs)
FastMCP over low-level Server

Phase 1 tools are all schema-friendly Python functions. The decorator's auto-schema from type hints maps cleanly to every tool we've roadmapped. The reflection above shows what we would have hand-written without FastMCP — that's the cost we're avoiding.

Subpackage mcp/ over flat mcp_server.py

YAGNI said go flat. We went subpackage anyway because the roadmap already names the future modules (tools/campaigns.py, tools/screens.py, tools/heatmap.py). When the future is concrete, pre-structuring isn't speculation — it's preparation. The cost is one extra __init__.py.

stdio first, streamable-http later (M0.4b)

The grill-with-docs skill caught that pure-SSE transport is being phased out in the MCP spec in favor of streamable-http (which still uses SSE framing under the hood for streaming responses). So we get the SSE-on-the-wire learning either way. stdio first keeps the M0.3 chunk focused on the protocol; M0.4b — a new chunk we inserted — will be a single-line transport switch (mcp.run(transport="streamable-http")) with its own reflection page comparing wire behavior side-by-side.

Script entry over python -m

[project.scripts] is idiomatic Python packaging. uv run dooh-mcp reads like a deliberate command. The Claude Desktop config (M0.4) becomes a shorter, cleaner block. And both python -m dooh_control_tower.mcp.server and python server.py still work — defensive triple-entrypoint at no extra cost.

No print() in the MCP module

Stdio MCP is stdout. Any stray print() corrupts the JSON-RPC frame stream and silently breaks the server. The one-line comment at the top of server.py is a tripwire for future-self. Logging will go to stderr.

▸ Workflow notes (skills that fired)
Skill used: grill-with-docs (~6 questions, ~15 min) before any code was written.
What it caught that I'd have missed:
  • Pure-sse transport is deprecated in the MCP spec; streamable-http is the modern path and still uses SSE framing. Resulted in inserting M0.4b as a dedicated transport-comparison chunk instead of fighting the deprecated path.
  • Calibration: when the future is named on the roadmap, pre-structuring (subpackage) isn't YAGNI — it's planning. Recorded as a memory signal for future judgment calls.
  • Defensive triple-entrypoint pattern (script + module + direct file) costs nothing and removes a class of "but how do I run this?" friction.
What it didn't catch: the mcp/ subpackage name shadows the SDK's top-level mcp package visually. Python 3's absolute-import rules resolve correctly (from mcp.server.fastmcp … still finds the SDK), but it's a footgun for readers. Worth a future ADR if it ever bites.
