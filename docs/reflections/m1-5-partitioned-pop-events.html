First time-series — partitioned PoP events

The domain stops being snapshots and starts being a stream. pop_event is the impression store — one row per render, partitioned by day, dimensioned to handle ~35M rows at portfolio scale. pacing_bucket is the hourly aggregate M6.1 will live on. The first grill-with-docs session for a schema chunk produced one ADR, one CONTEXT.md, and a denser-than-usual decision tree.

Two write paths: synchronous ad-server writes pop_event one row at a time (M3.4); a periodic job projects pop_event aggregates into pacing_bucket (M6.1). Same table, two consumers — same separation the rest of the codebase already follows (pure generator vs async upsert).

TL;DR — what's new

  • First declarative partitioning — PARTITION BY RANGE (event_date), daily children created on demand.
  • First natural-key composite PK — no surrogate UUID. ON CONFLICT DO NOTHING gives free retry-idempotency.
  • First denormalized column on principle — h3_r8 / h3_r9 on pop_event for point-in-time correctness, not just speed.
  • First migration with hand-written DDL — autogen captured postgresql_partition_by cleanly; only the default partition needed an op.execute().
  • First ADR (ADR-0001) — the three coupled choices above are hard-to-reverse and worth documenting.
  • First CONTEXT.md — glossary nailing down PoP, impression, pacing bucket, H3 cell, targeting polygon, daypart, eligibility.

Partition granularity ≠ data resolution

This was the key clarification during the grill. The two get conflated:

ConceptWhat it controlsOur choice
Partition granularityHow Postgres physically splits rows into child tables. Drives retention (DROP PARTITION) + partition pruning at scan time.Daily — one child per day.
Data resolutionThe grain of one row — what's recorded in each event.Per-impression, with event_ts timestamptz precise to the microsecond.

So daily partitions don't lose hourly data. Every row inside the daily partition still has its full event_ts. The M4 heatmap groups by date_trunc('hour', event_ts) inside whatever partitions get scanned. Partition pruning eliminates days outside the range; the GROUP BY does the hour-level rollup inside those scanned days.

Why hourly partitioning would be wrong here: 1,440 child tables (24h × 60d), tiny ~24K-row partitions where the catalog cost outweighs the pruning win, no retention pattern that wants "drop Tuesday 2pm." DOOH retention is per-day. Partition cadence should match retention cadence — that's the rule.

Why h3 is on the impression, not just the screen

The "screens have geom; H3 is derived; JOIN at query time" intuition is the wrong one for an impression store. Two reasons:

  • Point-in-time correctness. An impression's true location is where the screen was when it played. If a kiosk relocates, historical pop_events must keep their original H3. JOINing to the current screen row would silently rewrite history — wrong for any reporting downstream.
  • Hot-path JOIN-free aggregation. M4 heatmap: SELECT h3_r8, count(*) FROM pop_event WHERE event_date BETWEEN ... GROUP BY h3_r8. At 35M-row scale the JOIN cost is not theoretical.

~16 bytes/row × 35M = ~560 MB of denorm cost over 60 days. Negligible. ADR-0001 captures the full reasoning.

Partition pruning, demonstrated

The point of daily partitions is that WHERE event_date = ? lets Postgres scan one child table instead of the parent's full physical extent. Verified at ship:

EXPLAIN SELECT count(*) FROM pop_event WHERE event_date = '2026-06-03';

 Aggregate  (cost=18.76..18.77 rows=1 width=8)
   ->  Seq Scan on pop_event_2026_06_03 pop_event  (cost=0.00..18.75 rows=4)
         Filter: (event_date = '2026-06-03'::date)

No scan of pop_event_default, no scan of other dates. Just the matching daily partition. At 35M-row scale, this is the difference between a 50ms query and a 5-second one.

Workflow notes

  • First full grill-with-docs schema session. 7 questions, ~1 hour. Produced one ADR + one CONTEXT.md + the design summary. Worth it — the partition-vs-resolution confusion would've shown up in M4 otherwise.
  • Pre-PRD challenged my Q4 default. I'd queued composite (id, event_date) in the roadmap; the grill surfaced the natural-PK alternative via pre-PRD's intent. Better answer than my starting point. [[feedback]]: the queued roadmap recommendations aren't always right; the grill earns its keep.
  • Alembic surprise — autogen captured postgresql_partition_by. I'd planned a one-line hand-edit; turned out to be zero. SQLA 1.2.6+ feature works through to Alembic autogen output.
  • ensure_partition() contract. The Python helper paid for itself immediately — the seed wrote rows for 3 distinct days; without partitions they'd have landed in _default. M3.4 sync writes will use the same helper.

Go deeper

  • BRIN vs btree on time-series PK. At 580K rows/partition, btree wins; cross-over point for BRIN dominance.
  • Generated columns + IMMUTABLE casts. event_date GENERATED ALWAYS AS ((event_ts AT TIME ZONE 'UTC')::date) STORED would remove the Python-side discipline; trade-off on migration complexity.
  • Default partition trap. Why a default partition blocks attaching overlapping ranges; how prod systems handle it (alarm + drain).
  • pg_partman. What it automates beyond ensure_partition() — retention cadence, premake, sub-partitioning. Worth before M3.5 (high-QPS simulator)?
  • Composite-PK insert cost. 4-column PK vs single UUID — write-path B-tree maintenance compared at 35M-row scale.
  • Timezone discipline. Storing hour_ts UTC vs America/New_York. Trade-off for multi-region (which we don't model). The "evening prime time" boundary slips ~4h vs NYC wall clock — when does that bite?
  • The ADR pattern itself. When is a coupled decision better as one ADR vs three? ADR-0001 bundles three because they're meaningless apart; what would make me split them?
