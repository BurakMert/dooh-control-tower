Why this way

1. Why SQLAlchemy 2.0 async instead of raw SQL

The grill opened with the question that determines the driver: how does the app talk to Postgres?

My recommendation was raw SQL — PostGIS-heavy workloads read more clearly as raw queries, the schema is small, and the showcase narrative ("here's the SQL the agent runs") is cleaner. You picked SQLAlchemy 2.0 async instead. Reasons that point that way: Alembic is the gold standard for migrations, SQLA 2.0 Declarative + Mapped[T] gives typed models that pair well with Pydantic, and the SQLA ecosystem is what employers expect.

Knock-on: chose psycopg3 over asyncpg (next collapse). PostGIS access will need GeoAlchemy2 when we add geometry columns in M1.1 — flagged as a small future research moment.

2. Why psycopg3 over asyncpg

The deciding sentence from the SQLAlchemy 2.0 changelog:

The same psycopg database URL works with both create_engine() and create_async_engine(), with the corresponding sync or asyncio version of the dialect selected automatically.

Translation: one driver covers both the FastAPI async path and Alembic (which is sync). With asyncpg we'd need psycopg2 or psycopg3 installed alongside, just to make Alembic work.

The perf gap (asyncpg historically ~10-30% faster) is invisible at our workload — analytical queries and a synthetic PoP simulator that won't push thousands of QPS. We're paying ergonomic cost for raw throughput we don't need.

Counter-case: if M3 ad-serving load profile turns out to need every microsecond, we swap drivers by changing the URL prefix from postgresql+psycopg:// to postgresql+asyncpg://. The rest of the app code doesn't move.

3. Why a module-level engine + lifespan only owns disposal

SQLA's AsyncEngine is cheap to create — the pool is lazy. No connections open until first checkout. So creating it at module import time (rather than inside the lifespan) costs nothing and keeps the wiring simple.

The lifespan only owns shutdown: await engine.dispose(). That call is idempotent — safe to call from both the FastMCP stdio lifespan and the FastAPI HTTP lifespan even if a future refactor makes both fire in the same process.

Trade-off accepted: if you wanted per-request engine swapping for tests, you'd want the engine inside the lifespan instead. We're not there yet — when we get to integration tests in a later milestone, we'll likely use pytest-asyncio fixtures that create their own engine.

4. Why AsyncExitStack instead of nested async with

Naive way (works for two resources, scales badly):

async with mcp.session_manager.run():
    async with db_lifespan():
        yield

Stack-based way (what we shipped):

async with contextlib.AsyncExitStack() as stack:
    await stack.enter_async_context(mcp.session_manager.run())
    await stack.enter_async_context(db_lifespan())
    yield

Why bother now, with only two resources? Because we know what's coming: Redis for caching (Phase 2?), an HTTP client pool for the lite ad-server's downstream calls, maybe a background scheduler for the pacing rebalancer (M6.1). Nested async with walks off the right margin. AsyncExitStack stays flat.

Per memory feedback_yagni_calibration.md: when the roadmap names the future and the cost is low, prefer pre-structuring. This is exactly that shape.

5. Why pool_size=5, max_overflow=5

Single uvicorn worker. Agent-paced traffic — a handful of tool calls per minute, each with one DB round-trip. Five baseline connections is comfortably above that.

max_overflow=5 gives 10 max in-flight without raising errors during bursts. Postgres default max_connections is 100, so we're using a tenth of the budget — fine for dev, fine for the synthetic PoP simulator in M3 if it stays in a separate process.

Revisit at M3 when we add the ad-serving path (/ad endpoint) — that's the first time we'll have concurrent request load against the DB. Numbers to watch: pool wait time, P99 query latency, postgres pg_stat_activity.

6. Why pool_pre_ping=True

Cheap SELECT 1 on every connection checkout. The cost is one extra round-trip per request (microseconds on localhost, sub-millisecond on a VPC). The benefit: stale connections — killed by a Postgres restart, a Docker pause, an idle timeout — get detected and replaced before they hand a broken connection to your code.

In a dev loop where you docker compose restart postgres mid-session, this is the difference between "next request works" and "next 5 requests fail with weird connection errors before the pool recovers."

7. Why postgis/postgis:17-3.5 instead of postgres:17 + manual extension

Three reasons:

  • The image's docker-entrypoint-initdb.d pre-creates the extension on first boot — no migration step needed to bootstrap PostGIS.
  • The build is the canonical PostGIS team build — GEOS, PROJ, GDAL, and pgRouting versions are known-compatible. Hand-rolling has a footgun: apt-get install postgis in a postgres image can pull a GEOS that mismatches the binary expectations.
  • Tag versioning (17-3.5) is two-axis: Postgres major + PostGIS minor. Pinning both axes is exactly what we want for reproducible dev.

Cost: amd64 only. Apple Silicon needs platform: linux/amd64 and runs under Rosetta. Acceptable for dev; prod can drop the pin on a Linux/amd64 host.

Workflow notes

grill-with-docs caught the sync+async one-driver win

Without checking docs, I would have defaulted to "asyncpg is faster, pick it." That's true on raw benchmarks but misses the operational story: Alembic is sync, your migration tooling needs a sync driver, and shipping two Postgres drivers is a footgun.

Context7's pull of the SQLA 2.0 changelog surfaced the exact sentence ("the same psycopg URL works with both…") that flipped the recommendation. Six minutes of doc query vs. several days of "why does Alembic need its own driver" confusion down the road.

Counted against the feedback_ai_workflow_habit.md ledger as a clean grill-with-docs payoff. Commit footer: Skills: grill-with-docs, handoff.

Why one combined reflection for M0.6a + M0.6b

M0.6a alone was pure plumbing — a docker-compose file. Per feedback_visual_reflections.md, pure plumbing skips the reflection.

But the M0.6a decisions (image choice, Apple Silicon Rosetta pin, env file split) matter enough to capture once. Bundling them into the M0.6b reflection — the chunk that actually introduces concepts (async driver story, pool, lifespan composition) — keeps the M0 documentation set coherent.
