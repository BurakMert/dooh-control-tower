Reflection · M1.4

First foreign keys — campaign, creative, targeting

The domain stops being a flat list of screens and starts being a graph. Three new tables, two new relationship shapes (1:N and 1:1), one new geometry type (MULTIPOLYGON), and the first multi-table seed orchestration. The agent can now describe a campaign — and from M3 onward, decide which screens it's eligible to run on.

Big picture

erDiagram campaign ||--o{ creative : "1:N (cascade)" campaign ||--o| targeting : "1:1 (cascade)" screen }o..o{ targeting : "spatial (ST_Contains)" campaign { uuid id PK string external_id UK string advertiser string state date start_date date end_date int daily_budget_cents string pacing_strategy } creative { uuid id PK uuid campaign_id FK string external_id UK string creative_type int width int height } targeting { uuid id PK uuid campaign_id FK_UK geometry geom "MULTIPOLYGON 4326" int daypart_start_hour int daypart_end_hour } screen { uuid id PK geometry geom "POINT 4326" string market }

Two relational edges (FK), one spatial edge (PostGIS). Real DOOH platforms reuse creatives across campaigns (N:N), but for portfolio scope 1:N is honest enough — promote in M5 if it actually bites. screen → targeting is dotted because it's a runtime predicate (ST_Contains), not a stored edge.

TL;DR — what's new vs M1.1

  • First foreign keys — ON DELETE CASCADE from creative and targeting back to campaign. Drop a campaign, the dependents go too.
  • First 1:1 relationship, encoded as UNIQUE on targeting.campaign_id + relationship(..., uselist=False) on the ORM side.
  • First multi-table autogen. The include_object filter from M1.1 kept TIGER tables out of the diff — clean 60-line migration instead of last time's 700-line nightmare.
  • First MULTIPOLYGON column. POLYGON would suffice today; MULTIPOLYGON future-proofs disconnected coverage (paste-in polygons in M5.4).
  • First two-phase async seed. Insert parents → re-read IDs → wire children. Avoids string-keyed FK gymnastics.

The JSONB-vs-table tension

The roadmap explicitly flagged this. JSONB is real-world tempting: campaign.targeting as one JSONB blob = no join, schema-flex, easy to evolve. We didn't pick it. Reasoning:

  • Spatial indexing. GiST works on geometry columns. JSONB-stored GeoJSON has to be deserialized + re-cast (ST_GeomFromGeoJSON(t.targeting->'geom')) per row — kills the index.
  • FK leverage. A separate row means cascade rules, partial uniqueness, future foreign keys (e.g., link a targeting preset). JSONB is opaque to the relational layer.
  • Schema observability. \d targeting shows shape; \d campaign with a JSONB column shows nothing about what's inside.

Where JSONB would win: if targeting were truly polymorphic (geo, contextual, audience, behavioral all in one blob) and the read pattern were "fetch the whole campaign as one document." Neither is true here.

Two-phase async seed

Children need campaign_id FKs, but the campaign UUIDs are server-generated. The seed handles this in two passes inside one session:

1. INSERT campaigns ON CONFLICT DO NOTHING  ← idempotent
2. SELECT id, external_id FROM campaign      ← resolve mapping
   WHERE external_id IN (...)                  (picks up existing too)
3. INSERT creatives + targeting               ← FK-wired
   ON CONFLICT DO NOTHING

The pure generator (_generate_campaign_rows) carries a placeholder _campaign_external_id on each child row. The async seeder pops it and substitutes the resolved UUID via _attach_campaign_fk. Separation of concerns: the generator stays unit-testable; only the FK-wiring step touches the DB.

Why bounding boxes (not shapely)

Each campaign's targeting polygon = padded bbox around 3-8 sampled screens. Hand-rolled WKT, no shapely dep. Verified the polygons aren't trivially empty:

SELECT c.external_id, count(s.id) AS hits
FROM campaign c
JOIN targeting t ON t.campaign_id = c.id
LEFT JOIN screen s ON ST_Contains(t.geom, s.geom)
GROUP BY c.external_id;
-- 19–48 screens per polygon. M3.3 eligibility sets will not be empty.

The bbox + buffer approach over-covers the sampled set (intentional — buffers screens near the edge inside) and under-models true campaign coverage (which is rarely rectangular). Good enough for the seed; M5.4 will introduce paste-in polygons for real shapes. Adding shapely now would have been [[feedback_yagni_calibration]] on the wrong side — the future is named, but PostGIS-side ST_Buffer(ST_ConvexHull(...)) covers it without a Python dep.

Go deeper

  • 1:1 in SQLA — uselist=False vs scalar_relationship. When does it matter for query planning?
  • Cascade semantics — DB-level ON DELETE CASCADE vs SQLA-level cascade="all, delete-orphan". We have both; when do they diverge?
  • POLYGON vs MULTIPOLYGON — type coercion rules in PostGIS, why ST_Multi exists, and when to use a GEOMETRY generic column with a CHECK.
  • RNG stream isolation — when adding a second seeded random stream, should we use a single rng + jumpahead (numpy) or two independent Random(seed_n)? Trade-offs.
  • Two-phase insert pattern — could RETURNING id on the parent INSERT eliminate the SELECT? (Yes — with psycopg3 and SQLA Core; design tax was minor.) Revisit if seed scale grows.
  • Budget representation — cents-as-int vs numeric(12, 2) vs Postgres money. Which is the standard for ad-tech / fintech, and why does the answer differ?
