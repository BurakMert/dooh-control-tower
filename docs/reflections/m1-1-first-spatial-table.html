How it works — model to table

The model declaration in screen.py:

class Screen(Base):
    __tablename__ = "screen"

    id: Mapped[UUID] = mapped_column(
        Uuid(as_uuid=True),
        primary_key=True,
        server_default=func.gen_random_uuid(),
    )
    external_id: Mapped[str] = mapped_column(unique=True)
    name: Mapped[str]
    geom: Mapped[WKBElement] = mapped_column(
        Geometry("POINT", srid=4326, spatial_index=True),
    )
    screen_type: Mapped[str]
    resolution_width: Mapped[int]
    resolution_height: Mapped[int]
    market: Mapped[str]
    is_active: Mapped[bool] = mapped_column(server_default=text("true"))
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), onupdate=func.now()
    )

Compiled by Alembic + psycopg3 → Postgres DDL:

                                  Table "public.screen"
      Column       |           Type           |      Default
-------------------+--------------------------+-------------------
 id                | uuid                     | gen_random_uuid()
 external_id       | character varying        |
 name              | character varying        |
 geom              | geometry(Point,4326)     |
 screen_type       | character varying        |
 resolution_width  | integer                  |
 resolution_height | integer                  |
 market            | character varying        |
 is_active         | boolean                  | true
 created_at        | timestamp with time zone | now()

Indexes:
    "screen_pkey" PRIMARY KEY, btree (id)
    "idx_screen_geom" gist (geom)
    "screen_external_id_key" UNIQUE CONSTRAINT, btree (external_id)

Three things to clock: (1) Mapped[UUID] + Uuid() → native uuid column with server-side default. No client-side ID generation. (2) Geometry("POINT", srid=4326, spatial_index=True) → geometry(Point,4326) column and a GiST index in one declaration. Spatial containment queries in M3 will be O(log n) instead of O(n). (3) DateTime(timezone=True) → timestamp with time zone, so now() stores UTC explicitly. Without it you'd get a naïve timestamp and timezone bugs in M4 reporting.

Why this way

1. Why Alembic on day one (not create_all)

M1 adds five tables. Each addition is a schema change. create_all means dropping and recreating — fine for a spike, terrible the moment you have data you don't want to throw away (which will be true by M1.2).

The setup tax was real (~15 min: alembic init, rewrite env.py, fight include_object). The interest payments are: every M1.x onward is alembic revision --autogenerate -m "..." + a review pass. Five minutes per migration vs. drop+rebuild gymnastics.

Pre-paid because today is fresh-day. Cheaper than "Alembic later, under deadline."

2. Why this include_object filter — the autogenerate trap

First autogenerate ran came back with a 732-line migration trying to drop 30+ tables we'd never heard of.

Cause: the postgis/postgis:17-3.5 image ships the TIGER geocoder. Tables land in the tiger schema, but the default Postgres search_path includes "$user", public, topology, tiger. SQLA's inspector pulls every reachable table. Alembic compares against our Base.metadata (which has one table) and concludes the rest must be removed.

The fix is a custom include_object hook:

def include_object(object, name, type_, reflected, compare_to):
    if not alembic_helpers.include_object(object, name, type_, reflected, compare_to):
        return False
    # If it exists in the DB and isn't in our metadata, we don't own it.
    if reflected and compare_to is None:
        return False
    return True

Read as: "we manage what we declared; everything else is someone else's problem." The general rule for any project sharing a database with extensions, sidecars, or migration-tool overlap.

Bookmark for M3+ if we ever introduce a second app with its own tables in this DB — that rule will still hold.

3. Why server-side UUID default (not Python-side)

Two viable patterns for UUID PKs in SQLA 2.0:

  • Python-side: mapped_column(default=uuid.uuid4). ID assigned at session.add() time. Pros: ID known before insert. Cons: ID generation logic spread across app code.
  • Server-side (chosen): server_default=func.gen_random_uuid(). ID assigned by Postgres. Pros: single source of truth, no Python import needed, bulk INSERTs don't need Python to generate N UUIDs. Cons: ID only known after RETURNING.

Server-side wins for this project because (a) we'll never insert from anywhere except this app + the M1.2 generator, (b) Postgres' gen_random_uuid() uses the kernel RNG (better entropy than Python's uuid4 in some environments), (c) SQLA's RETURNING integration means the model still has the ID after session.add(); await session.flush().

Why not uuidv7? It's time-sortable (better B-tree locality). But Python's stdlib uuid.uuid7() doesn't land until 3.14, and Postgres needs an extension or trigger. Not worth the dep yet — we'll revisit if the pop_event table (M1.4) shows hot-path insert latency.

4. Why spatial_index=True in the column declaration

GeoAlchemy2's Geometry(..., spatial_index=True) emits both the column and a GiST index in one declaration. The index is the only thing that makes ST_Contains, ST_DWithin, and other PostGIS predicates fast.

Without it, M3.3 (geo targeting) would be a full table scan per ad request. With it, the GiST tree narrows candidate screens to a handful before the exact predicate runs.

The index is named idx_screen_geom by GeoAlchemy2's convention. Alembic emits CREATE INDEX ... USING gist (geom) in the migration — visible in \d screen as "idx_screen_geom" gist (geom).

5. Why SRID 4326 (and not Web Mercator 3857)

SRID 4326 = WGS 84, the standard lat/lon coordinate system (the one GPS speaks). Pros: universal, what every input feed will give us, what every consumer (Mapbox, Leaflet, mcp-ui) expects.

SRID 3857 (Web Mercator) is what tile renderers use internally — it's projected onto a 2D plane for display. Storing as 3857 would be premature optimization: PostGIS converts 4326 ↔ 3857 in microseconds, and storing as 3857 would force us to convert on every input/output. 4326 is the language of input; 3857 is the language of pixels. Store in the input language, project only when rendering.

One thing to know: PostGIS' ST_DWithin on 4326 takes distance in degrees, not meters, unless you cast to geography first. We'll handle that explicitly in M3.3 — flagged here so it doesn't sneak up.

6. Why screen_type as str, not a Postgres ENUM

Tempting to make screen_type a typed enum ('billboard' | 'street_furniture' | 'transit' | 'mall') on day one. Resisting because:

  • We don't yet know the closed set. M1.2 (synthetic generator) and M3 (targeting) are what will reveal which types matter.
  • Postgres ENUM is hard to evolve. Adding a value is one ALTER. Removing or renaming is a multi-step migration.
  • str → enum is a one-line migration when we're ready. enum → str is painful.

"Easy to tighten, painful to loosen." Tighten in M3 when the cardinality is settled.

Workflow notes

grill-with-docs surfaced the migration-strategy decision early

The grill opened with "Alembic now vs create_all vs raw SQL?" — a decision I'd have skipped past instinctively (Alembic feels obvious in retrospect). Naming it explicitly meant I considered the counter-case (create_all as legitimate for a spike) instead of cargo-culting Alembic.

The GeoAlchemy2 docs pull confirmed Geometry("POINT", srid=4326, spatial_index=True) still maps cleanly onto SQLA 2.0's Mapped[WKBElement] — saved a tangent into "is GeoAlchemy2 still maintained?" land.

When autogenerate goes wrong: trust your metadata, not the database

First autogenerate produced a 732-line migration trying to drop 30+ TIGER tables. Instinct could go three ways:

  1. Edit the migration by hand to remove the drops (brittle — next autogen does it again).
  2. Switch to hand-written migrations (defeats the point of choosing Alembic).
  3. Fix include_object so autogenerate stops seeing what we don't own (right answer).

This rule generalizes: autogenerate is a function from (your metadata, current DB) to diff. When the diff is wrong, the fix is almost always to constrain what counts as "your metadata," not to manually paper over the output.
