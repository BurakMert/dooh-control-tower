How it works

The anchors carry market and act as the geographic centroids:

ANCHORS = (
    Anchor("Manhattan", "Times Square",     -73.9855, 40.7580),
    Anchor("Manhattan", "Penn Station",     -73.9935, 40.7506),
    Anchor("Manhattan", "Grand Central",    -73.9772, 40.7527),
    # … 9 more …
    Anchor("Queens",    "Flushing",         -73.8326, 40.7596),
)

The trick is balancing the anchor weights so per-market totals converge to the desired share:

market_anchor_counts = {m: sum(1 for a in ANCHORS if a.market == m)
                        for m in MARKET_SHARE}
anchor_weights = [MARKET_SHARE[a.market] / market_anchor_counts[a.market]
                  for a in ANCHORS]
# Manhattan (7 anchors, 60% share) → each weight = 0.60 / 7 ≈ 0.0857
# Brooklyn  (3 anchors, 25% share) → each weight = 0.25 / 3 ≈ 0.0833
# Queens    (2 anchors, 15% share) → each weight = 0.15 / 2 = 0.0750

Then for each screen: pick an anchor by weight, jitter ±0.005° (~500m), pick a screen_type and resolution by weight. Pure-function, deterministic, no side effects.

Insertion is one batched executemany:

stmt = pg_insert(Screen).on_conflict_do_nothing(
    index_elements=["external_id"]
)
await session.execute(stmt, rows)
await session.commit()

Idempotent for free: rerun the script, the conflict on external_id drops duplicates silently. Want a clean regenerate? TRUNCATE screen first.

Why this way

1. CLI as a stand-in for the production CSV importer

The agent doesn't generate screens — in a real DOOH network, screens exist physically and arrive via a feed from the network operator (CSV, JSON, vendor API). The right shape for that work is a CLI/job, not a chat tool.

Modeling the seed as uv run dooh-seed rather than an MCP tool keeps the demo honest about its analogue. The reflection can say: "in production this becomes a CSV importer pointed at the network's feed; the shape stays the same."

Bonus: M1.3 (campaigns), M1.4 (pop_event), and M1.5 (H3 cells) will all need seed paths. One CLI structure (seed.py with sub-functions per table) compounds; five MCP tools doing dev-loop work would be five wrong shapes.

2. Why hand-curated anchors over gaussian clustering

Three plausible algorithms; only one reads cleanly:

  • Pure uniform over a bounding box — looks like random snow on a map, not a real network.
  • Gaussian clustering — pick K centroids, sample with sigma. Mathematically clean, hard to reason about ("is sigma=0.003 a city block or a borough?").
  • Anchors + uniform jitter (chosen) — hand-curated lat/lon for 12 known places, ±0.005° (~500m) uniform jitter. The jitter has a city-meaningful bound; the anchors carry semantics (Times Square ≠ Wall Street).

The anchors also drive market and seed plausible names ("Times Square Screen #003"). One declarative tuple → three columns' worth of plausibility.

3. Why fixed seed (and why random.Random(seed) not random.seed())

Reproducibility is the killer feature for fixtures. Same seed → same 100 screens. Tests in M3 can hard-code expected external_ids and not flake.

random.Random(42) creates a *private* RNG. random.seed(42) mutates the global RNG, which is shared with anything else in the process that calls random. The private form is contained — no spooky action at a distance if the test harness or a future module also touches randomness.

This is one of those "best-practice the codebase will outgrow if I don't write it this way now" choices. Cheap insurance.

4. The WKT POINT(lon lat) footgun

Well-Known Text (WKT) uses longitude first. So a screen near Times Square is POINT(-73.9855 40.7580), not POINT(40.7580 -73.9855).

This is the opposite of how every other format orders coordinates:

FormatOrderTimes Square
WKT (PostGIS)lon latPOINT(-73.9855 40.7580)
GeoJSONlon lat[-73.9855, 40.7580]
Google Maps / iOSlat,lon40.7580,-73.9855
JS Mapbox / Leafletlat,lng{lat:40.7580, lng:-73.9855}

The dataclass field naming is the defense in depth — Anchor(lon=..., lat=...) means you can't mix them up positionally. The WKT format string is the one place where order matters and there's no compiler help. Visible scrutiny at the format boundary, plus comments calling out the trap.

5. Why SQLA Core bulk over session.add_all and over psycopg3 COPY

At N=100, the three are subsecond. Performance isn't the differentiator:

PathApprox ms @ N=100Right when
session.add_all~50You want the ORM identity map / cascades / event hooks
session.execute(insert, [...])~20One-shot scripts, batches up to ~10k
psycopg3 COPY FROM STDIN~310k–1M+ rows in a hot loop

Bulk insert wins on readability for a one-shot script: no session lifecycle gymnastics, no identity map allocated for rows we won't mutate. The pg_insert variant also gives us ON CONFLICT DO NOTHING — idempotency in one line.

COPY waits for M3.5 — the synthetic PoP simulator pushes 10k+ events/sec, and that's where COPY's wire-level efficiency actually pays off. The M1.1 go-deeper bookmark stays alive there.

6. ON CONFLICT DO NOTHING vs full upsert vs truncate-and-rebuild

The seed needs an idempotency story. Three options:

  • ON CONFLICT DO NOTHING (chosen) — rerun, existing rows untouched, new rows added if external_ids are new.
  • ON CONFLICT DO UPDATE (upsert) — rerun overwrites existing rows. Useful if the generator changes (anchor moves, jitter changes) and you want the database to follow.
  • TRUNCATE first, then INSERT — clean slate every run. Loses any FK-linked downstream data.

Picked DO NOTHING because: (a) external_ids are stable and deterministic from the seed, (b) once campaigns/PoP events exist (M1.3+) they'll reference screens by id — overwriting screens would mean dangling FKs, (c) for the "I really want to regenerate" path, manual TRUNCATE screen CASCADE is the explicit choice the user should make.

Workflow notes

Bundling related decisions in the grill

The roadmap named four decisions for this chunk. I bundled randomness + clustering into one "synthetic data richness" tier question (Minimal / Anchored / Faked) instead of asking them separately.

Logic: they aren't independent — the right answer to "should we use Faker" depends on the same plausibility budget as "should we cluster around anchors." Asking them serially would have made the user think the same thought twice. Bundling kept the grill at 3 questions instead of 4 while preserving the choice structure.

Note for future chunks: when two decisions share an underlying axis, present them as tiers along that axis. When they're orthogonal, keep them separate.

The M1.1 bookmark on COPY paid off here

M1.1's go-deeper #5 was "psycopg3 COPY vs SQLA bulk insert" — bookmarked as a future deep dive. M1.2 made the decision concrete: SQLA bulk now, COPY later.

The bookmark didn't get pulled today (no reason to learn COPY here), but it stayed live and informed the M3.5 plan. Reflection bookmarks are themselves a kind of memory — they keep the workflow honest about what's been deferred vs forgotten.
