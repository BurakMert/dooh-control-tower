H3 indexing — the chunk that got smaller as it got sharper

M1.6 started as five queued decisions and ended up about half the original scope after the grill. The principle that did most of the trimming: the database stays a database. Three "let's add this" temptations got cut — h3-pg extension, h3 columns on screen, speculative btree indexes. What shipped: one Python dep, one ~5-line helper, one ~30-line backfill, one ADR.

Three writers, one helper, one column pair. screen.geom stays the single source of truth for location; pop_event gets its impression-time h3 snapshot at INSERT (ADR-0001) computed by the Python helper (ADR-0002).

TL;DR — what shipped

  • Dep: h3==4.5.0 via uv add h3.
  • Helper: compute_h3_cells(lat, lng) → (r8_int, r9_int). Two h3.latlng_to_cell calls + str_to_int for BIGINT storage.
  • Write-path integration: seed_pop_events() sets h3_r8/h3_r9 at insert from each sampled screen's lat/lng.
  • Safety net: backfill_pop_event_h3() — UPDATEs any pop_event row where h3_r8 IS NULL. Used to populate the M1.5-seeded rows; future M3.4/M3.5 writes set h3 at INSERT so the backfill is a no-op.
  • Reading discipline: M4 heatmap reads pop_event.h3_r8 directly — never JOINs screen for h3.
  • No new tables, no new columns, no new indexes. The M1.5 schema already had the columns. The migration count for this chunk: zero.
  • ADR-0002 documents the Python-not-extension choice plus the two adjacent "no" decisions (no screen-side h3, no speculative indexes).

Two kinds of "indexing"

The single biggest moment in the grill — the word "indexing" carries two meanings, and they're easy to conflate:

"H3 indexing""Postgres indexing"
What it meansComputing H3 cell IDs from (lat, lng, resolution) so a point can be looked up in a hexagonal grid.Adding btree / GiST indexes on table columns so queries don't sequential-scan.
This chunkYES — that's the whole job.NO — deferred to M4.1.
The trapPre-emptively adding (h3_r8, event_date) as a btree is the obvious move only if you confuse the two. We don't have the M4 heatmap query yet — adding a btree without knowing the actual query shape risks the wrong leading column. At ~10K-distinct-h3-value cardinality the planner often picks HashAggregate regardless, making the index irrelevant for the bulk-grouped case.

The discipline: index when you know the query AND can measure. M4 will have both. M1.6 has neither.

Does a Postgres btree help GROUP BY?

The question that triggered Q5's revision. Three ways an index helps GROUP BY:

  • Sorted-input GroupAggregate. If the index column order matches the GROUP BY columns, Postgres reads the index in order and emits groups without a sort step.
  • Index-only scan. If every column the query touches is in the index, the planner skips the heap entirely. Less I/O.
  • Selective filter. WHERE h3_r8 IN (viewport) narrows the scan before GROUP BY runs.

The catch: at ~10K-distinct-group cardinality, HashAggregate usually wins. It builds a hash table in one sweep, doesn't care about input order, fits in memory easily. So for the bulk M4 query SELECT h3_r8, count(*) FROM pop_event WHERE event_date BETWEEN ... GROUP BY h3_r8, the index is often skipped by the planner regardless.

Where an index genuinely wins: viewport-filtered queries (WHERE h3_r8 IN (...)) where the filter is the leading column. M4.5 (drill-down on hex click) is that shape — and that's the milestone where we'll EXPLAIN, measure, and add the right composite.

Go deeper

  • h3-pg adoption path. What would trigger us to actually add it? Most likely candidates: M3.3 polygon-to-hex eligibility (h3_polygon_to_cells), M4.3 hex outline rendering server-side (h3_cell_to_boundary_geometry), or any neighbor-of-cell traversal at scale.
  • HashAggregate vs GroupAggregate planner cost model. When does the planner actually pick GroupAggregate over HashAggregate? work_mem tuning, cardinality estimation, INDEX-only-scan visibility-map state.
  • UPDATE on a partitioned table. Our backfill UPDATE works across daily partitions transparently. What's the cost vs INSERT? When does CONCURRENT UPDATE become a problem?
  • h3 4.x API: default str vs memview_int. We use latlng_to_cell + str_to_int; could switch to h3.api.memview_int.latlng_to_cell for direct int return. Bench at simulator scale (M3.5).
  • Composite key bind-param UPDATE. Our backfill issues one UPDATE per row — fine at 30 rows, would batch at scale. The SQLA pattern: update().where(col == bindparam(...)).values(...) + session.execute(stmt, list_of_dicts). Try it next time we need bulk UPDATE.
  • Drift surface for derived data. If we ever add UPDATE screen SET geom = ..., what discipline ensures the corresponding pop_events' h3 stays correct? (Answer: per ADR-0001, they don't update — they preserve impression-time location.)
