First mcp-ui surface — five walls and a verified render

A Leaflet map of 100 NYC screens, rendered inline in an MCP host. The map itself was the easy part. Getting it to actually render took walking through five distinct walls in the mcp-ui spec — and then discovering that Claude Desktop's render is broken in stock builds, while MCPJam Inspector renders it correctly. This reflection is half the implementation, half the forensic record.

TL;DR

Build path required all five of these to render in spec-compliant MCP hosts:

  1. Tool decorated with meta={"ui": {"resourceUri": "ui://..."}} (MCP Apps SEP-1724 signal).
  2. Resource registered via @mcp.resource(uri, mime_type="text/html;profile=mcp-app") — the ;profile=mcp-app modifier is what hosts treat as "this is a UI surface, not plain HTML."
  3. HTML imports @modelcontextprotocol/ext-apps@^1.7.0 and calls App.connect({strict: true}) — without this, hosts iframe the HTML but the handshake never completes.
  4. Resource meta carries "mcpui.dev/ui-preferred-frame-size": ["100%","560px"] — without it, some hosts size the iframe at 0px.
  5. HTML uses height: 100vh + min-height instead of height: 100% — defends against parent-iframe height collapse.

Verification: renders cleanly in MCPJam Inspector (npx @mcpjam/inspector). Does not render in stock Claude Desktop despite the host recognizing the tool as Interactive and injecting "widget rendered" into the agent context. Bug filed in the project log.

Every arrow is a contract. Miss any one (wrong MIME, missing _meta, no App.connect(), no frame size) and the host either rejects the resource, accepts it but shows it as text, iframes it at 0px, or iframes the HTML but never finishes the handshake. We hit each failure mode in sequence.

The five walls — symptoms and fixes

WallSymptomFix
1. Inline vs MCP Apps Tool returned UIResource inline in content[]. Claude Desktop showed it as a collapsed "Result" widget. Added meta={"ui": {"resourceUri": ...}} to tool. Registered HTML as @mcp.resource(). Modern hosts fetch via resources/read.
2. MIME modifier "Unsupported UI resource content format" error. The Python SDK ships plain text/html. Override to text/html;profile=mcp-app via the mime_type kwarg on @mcp.resource() and by mutating ui.resource.mimeType on the inline UIResource.
3. No handshake Resource accepted, but HTML shown as code text. Host iframed it, waited for handshake, gave up. HTML imports @modelcontextprotocol/ext-apps@^1.7.0 and calls App.connect({strict:true}). Older postMessage('ui-lifecycle-iframe-ready') kept as fallback.
4. 0px iframe Agent narrated "Here's the map!" and host injected "widget rendered" into context, but the iframe was invisible (0 effective height). Added "mcpui.dev/ui-preferred-frame-size": ["100%","560px"] to resource meta.
5. CSS height collapse Defensive: even with a 560px iframe, html, body { height: 100% } evaluates to 0 if the parent doesn't set explicit height. Switched to height: 100vh; min-height: 560px. Added map.invalidateSize() after the next frame so Leaflet recomputes after host resize.

How it works — the wire shape

Tool listing (tools/list) — the discovery hook:

{{
  "name": "show_screen_map",
  "description": "Render the screen network on a Leaflet map...",
  "inputSchema": {{ "properties": {{}}, "type": "object" }},
  "_meta": {{
    "ui": {{ "resourceUri": "ui://dooh-control-tower/screen-map" }}
  }}
}}

Tool call (tools/call) — text content only, no embedded UI:

{{
  "content": [{{
    "type": "text",
    "text": "Rendering the screen network map (100 screens across Manhattan, Brooklyn, and Queens)."
  }}],
  "isError": false
}}

Resource read (resources/read) — the actual UI:

{{
  "contents": [{{
    "uri": "ui://dooh-control-tower/screen-map",
    "mimeType": "text/html;profile=mcp-app",
    "meta": {{
      "mcpui.dev/ui-preferred-frame-size": ["100%", "560px"]
    }},
    "text": "<!doctype html>...26KB of self-contained map HTML..."
  }}]
}}

Two round-trips per UI surface: tool call (data + UI reference), then resource read (the HTML itself). Cost: one extra hop. Benefit: the HTML never enters the model's context window, only the user's view.

Why this way

1. Why pull mcp-ui forward from M2.4/M2.5

Two reasons that turned out to be even more right than expected:

  1. Pipeline validation. If we'd hit the five walls in M2.4 under "third UI chunk in three days" pressure, it would have been brutal. Discovering the full staircase now, when M1.3 has the only timeline that allows real research detours, means M2.4 and M2.5 are mechanical reskins.
  2. Motivation. M1.1 and M1.2 were schema and seed. M1.3 gave us the first "wow" moment — 100 dots on a real map, in real time, after a long Postgres grind.

The cost: M1 grew from 5 to 6 chunks AND this one took ~3 pomodoros instead of 1. Trade I'd make again — but next time the budget for "first-of-its-kind chunks" should be 2-3 pomodoros from the start, not 1.

2. Why rawHtml + text over externalUrl

externalUrl means the Resource carries a URL pointing at our FastAPI app; the host iframes that URL.

rawHtml + text means the resource contains the HTML literally. Host iframes it via srcdoc.

For "render a snapshot of the current network state," rawHtml wins on three axes:

  • Fewer moving parts. No FastAPI route, no CORS dance, no "is uvicorn reachable from the host's sandbox?" question.
  • Self-contained debugging. The HTML is in the resource read response. Save to a file, open in a browser — that's the failure-isolation tool that confirmed our HTML was right while Claude Desktop kept failing to render it.
  • Snapshot semantics. Each call returns current state. M1.2 re-seed → next call shows the new screens.

When to switch: M4.3 (heatmap with time slider). Live state, user-driven interaction, existing FastAPI surface.

3. Why Leaflet + CARTO Voyager over MapLibre + vector tiles

Leaflet is ~40KB raster-tile rendering via DOM. MapLibre is ~200KB vector tiles via WebGL. In a sandboxed iframe, the smaller-and-safer path wins for the first ship — and WebGL sometimes hits sandbox restrictions in unexpected ways.

CARTO Voyager Light over plain OSM: no-token, cleaner cartography. Same L.tileLayer API — different URL template.

MapLibre worth the weight in M4.3 — GPU rendering helps with H3 hex overlays.

4. Why two independent tools (list_screens + show_screen_map)

list_screens returns list[ScreenSummary] — the data shape M2.3 will need. show_screen_map returns the UI. Each does its own DB query.

  • Different intents. "How many billboards in Brooklyn?" → list_screens + filter. "Show me the network" → show_screen_map.
  • Decoupled evolution. M2.3 adds filter parameters to list_screens. The map shouldn't be coupled to those.
5. Why color by market, not by screen_type

Markets: 3 values, geographically clustered → 3 colored regions, instantly readable.

Screen types: 4 values, distributed within markets → mixed-color noise in every borough, no spatial signal.

Rule: the color dimension should align with the geographic dimension when both are present. screen_type shows up in the popup — that's where attribute info belongs when it has no spatial story.

6. JSON island over inline JS string 
// Inline JS — escape dance problem:
<script>const SCREENS = {{json.dumps(screens)}};</script>
// JSON island — chose this:
<script type="application/json" id="screens-data">{{screens_json}}</script>
<script>const SCREENS = JSON.parse(document.getElementById('screens-data').textContent);</script>

JSON-in-JS-string has subtle escape problems: </script> sequences in the data, backslashes, quotes. JSON island sidesteps all of it — script type isn't JS so the browser treats contents as opaque text; JSON.parse handles the parse.

7. height: 100vh over height: 100%

height: 100% requires the parent to have a defined height. If the host iframe is sized at auto, body { height: 100% } evaluates to 0. Then Leaflet computes its tiles into a 0-height container and the map "renders" invisibly.

height: 100vh is viewport-relative — 100% of the iframe's own viewport height, regardless of what the host set the iframe's CSS height to. Belt-and-suspenders with min-height: 560px.

Plus: requestAnimationFrame(() => map.invalidateSize()) after init, and window.addEventListener('resize', ...) after that, so Leaflet recomputes when the host eventually applies the preferred frame size.

Workflow notes

Browser preview as failure-isolation

After every code change, the resource was dumped to /tmp/screen-map-preview.html and opened in a browser. The map rendered cleanly there throughout — that's the diagnostic that told us the HTML was right while Claude Desktop kept failing.

Order of operations for any future mcp-ui chunk: (1) build HTML → (2) save to file → (3) open in browser → (4) then test in host. Each step is independently verifiable.

Pivoting hosts when the primary one fails

After ~4 rounds of "tweak server-side, retest Claude Desktop, still doesn't render," the right move was to test in a different host. MCPJam Inspector (npx @mcpjam/inspector@latest) renders the map cleanly via STDIO.

That single test converted hours of "is this our bug?" anxiety into definitive ground truth: our implementation is right; this host has a bug. Worth burning a pomodoro on second-host verification any time the primary host's failures are silent or ambiguous.

(Goose was the first second-host attempt — failed at brew tap. Lesson: pick the host with the lightest install for verification.)

grill-with-docs got 90%. The last 10% came from the dev community.

The opening grill picked the right Resource shape and tool pattern. What it didn't surface: the ;profile=mcp-app MIME modifier (older Ruby doc, dropped from Python SDK), the App.connect() handshake requirement (newer spec, no Python helper), and the preferred-frame-size hint (renders at 0px without it).

The MIME modifier came from a careful re-read of the docs after the "Unsupported UI resource content format" error. The App.connect() requirement came from a developer note the user found in a GitHub thread. Both required iterating against a real host's failures — docs alone weren't enough.

Add to the grill checklist for future host-binding chunks: "After my SDK code, do I also need to embed any client-side runtime (like ext-apps) for the host's handshake?" If yes, version compatibility between server and client runtime is the next thing to check.

First lessons-learned entry for M7.2 (Phase 2 motivation)

M7.2 (lessons-learned doc) is the "raw material for Phase 2 motivation" chunk. This is its first entry: even when the implementation is spec-correct, the host's render quality determines what the user sees. Phase 1's Claude Desktop dependency is a single point of failure for every visual surface. Phase 2 (Pydantic AI + A2UI + CopilotKit, own Next.js app) is exactly the architecture that removes that dependency.

This chunk converted "Phase 2 is interesting because we'll have more control" into "Phase 2 is necessary because Phase 1's host render is observably broken." That's a much stronger Phase 2 motivation, and it landed in the lessons-learned bucket six chunks early.
