DOOH Control Tower — Reflection M0.4b

stdio vs streamable-http: one FastMCP, two transports

M0.3 served the MCP server over stdio for Claude Desktop. M0.4b mounts the same FastMCP instance into our FastAPI app as a streamable-http endpoint at /mcp, so any HTTP client can talk to it. The interesting part isn't "we added a second transport" — it's how little had to change to do it.

Big picture

One FastMCP object owns the tool registry. Two transports plug into it. Claude Desktop launches us as a subprocess and speaks JSON-RPC over the stdin/stdout pipe. HTTP clients POST to /mcp on uvicorn and read SSE frames in response. Neither knows about the other.

The FastMCP instance is the source of truth for tools, resources, and prompts. Transports are how clients reach it.

What we did

  1. Added stateless_http=True and streamable_http_path = "/" to the FastMCP constructor in src/dooh_control_tower/mcp/server.py.
  2. In src/dooh_control_tower/app.py, imported the mcp instance and mounted mcp.streamable_http_app() at /mcp.
  3. Added a lifespan async context manager that runs mcp.session_manager.run() for the life of the uvicorn worker.
  4. Wrote scripts/probe-streamable-http.sh — three curl POSTs (initialize, tools/list, tools/call) that print raw SSE frames.
  5. Verified end-to-end: /health still returns JSON, the probe returns three clean SSE frames, stdio path remains unchanged (Claude Desktop still calls about successfully).

How the wire looks (side by side)

Same JSON-RPC payload, different transport envelopes. The pedagogical point of this chunk: see the envelope difference with your own eyes.

stdio (M0.3)
Newline-delimited JSON-RPC over the process pipe. One message per line. Connection is the process.
# client → server (stdin)
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

# server → client (stdout)
{"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2025-11-25","capabilities":{...},"serverInfo":{"name":"DOOH Control Tower","version":"1.27.2"}}}
streamable-http (M0.4b)
HTTP POST. Response is Content-Type: text/event-stream. Each event is event: NAME + data: PAYLOAD + blank line.
# client → server (HTTP POST /mcp)
POST /mcp/ HTTP/1.1
Content-Type: application/json
Accept: application/json, text/event-stream

{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

# server → client (SSE response body)
event: message
data: {"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2025-11-25","capabilities":{...},"serverInfo":{"name":"DOOH Control Tower","version":"1.27.2"}}}

The payload is identical JSON-RPC 2.0. stdio uses a line as the message boundary; SSE uses \n\n. SSE also gives the server a way to push more events on the same response (progress notifications, multi-part results) without a second HTTP round-trip. We don't use that yet — but stateless+SSE keeps the door open.

The code (the whole change)

src/dooh_control_tower/mcp/server.py
# stdio MCP wire uses stdout for JSON-RPC frames — never print() in this module.
from mcp.server.fastmcp import FastMCP

# stateless_http=True: no session IDs, every HTTP request stands alone. Flag is
# a no-op for stdio transport. streamable_http_path="/" avoids the default
# `/mcp` suffix when mounted (we mount at `/mcp` ourselves in app.py).
mcp = FastMCP(name="DOOH Control Tower", stateless_http=True)
mcp.settings.streamable_http_path = "/"


@mcp.tool()
def about() -> str:
    """Describe the DOOH Control Tower MCP server."""
    return (
        "DOOH Control Tower — a chat-driven operational tool ..."
    )


def main() -> None:
    mcp.run()  # stdio


if __name__ == "__main__":
    main()
src/dooh_control_tower/app.py
import contextlib

from fastapi import FastAPI

from dooh_control_tower.mcp.server import mcp


@contextlib.asynccontextmanager
async def lifespan(app: FastAPI):
    # FastMCP's session manager owns streamable-http connection
    # lifecycle — must run exactly once per process.
    async with mcp.session_manager.run():
        yield


app = FastAPI(title="DOOH Control Tower", version="0.1.0", lifespan=lifespan)


@app.get("/health")
def health() -> dict[str, str]:
    return {"status": "ok"}


app.mount("/mcp", mcp.streamable_http_app())

Total diff: 3 import lines, the lifespan, one mount call, two FastMCP constructor changes. The transport plumbing is FastMCP's job; we just composed two ASGI apps together.

Uvicorn mental model (why /health and /mcp coexist cleanly)

  • One event loop juggles both. Sub-millisecond /health calls interleave with long-lived SSE streams. As long as everything is async, neither starves the other.
  • Blocking code = self-DoS. A sync time.sleep or a sync DB call freezes the whole worker. Everything we write goes async-first.
  • Shared memory = shared state. The mcp instance, the session manager, FastAPI's app.state — all live in one process. That's why the lifespan runs once per worker.
  • Scaling later = --workers N. Stateless + SSE makes that safe — no session affinity to worry about.

Why this way (decisions and trade-offs)

Why mount into FastAPI instead of running a separate HTTP server?

Alternative: mcp.run(transport="streamable-http") spins up its own uvicorn on its own port.

Why mount instead: one process serves both /health and /mcp. Matches the eventual deploy shape (one container, one port). Demonstrates the ASGI mount pattern — a real-world skill we'll reuse for admin routes, status pages, etc. The cost is one lifespan context manager.

Roadmap test: M0.5+ adds tools that will share infra (DB connection pool, observability) with the rest of the app. Mounting is the natural shape for that.

Why stateless_http=True?

Stateful (default): server tracks a session ID per client. Required for server-initiated messages — progress notifications on long tool calls, resource subscriptions, sampling requests.

Stateless (our pick): every HTTP request is independent. No session memory. Recommended for scalability — any worker can serve any request.

Why now: none of our current tools push notifications. They take args, return a value, done. Stateful would be unused complexity.

When to flip: the moment we add a tool that streams progress or pushes "this campaign changed" events. Likely M5/M6.

Common confusion: "session" here is MCP protocol session, not application session. A campaign-being-edited still lives in the DB regardless.

Why streamable_http_path = "/"?

Without this override, streamable_http_app() serves at the path /mcp by default. If we mount that ASGI app at /mcp in FastAPI, the final URL is /mcp/mcp — ugly, and a footgun for anyone trying to remember the URL.

Setting streamable_http_path = "/" tells FastMCP "you're at the root of whatever's mounting you" — so the final URL is just /mcp.

grill-with-docs caught this — would have shipped /mcp/mcp by accident otherwise.

Why SSE responses (not JSON) for the streamable-http transport?

FastMCP supports json_response=True — one JSON blob per request, no streaming. Simpler over the wire.

Why we picked SSE anyway: the chunk goal was literally "observe SSE on the wire." Pedagogically, SSE is the more interesting wire shape because it's what enables MCP's server-push features. We get the SSE format now even though we don't yet push events — when we add notifications, the wire shape doesn't change, just what flows over it.

Why keep stdio at all if HTTP works?

Claude Desktop's MCP integration prefers stdio for local servers — no network, no port conflicts, no TLS to manage, automatic subprocess lifecycle (Claude Desktop owns the process). The cost of supporting both is approximately zero (one extra constructor flag), and the benefit is "works in both contexts" — Claude Desktop (stdio) and Pydantic AI / web (HTTP).

Workflow notes

What grill-with-docs caught on this chunk
  • The /mcp/mcp footgun. The default streamable_http_path is /mcp. Mounting that under /mcp would have given us a stuttering URL. Docs surfaced it; we set the override.
  • The mandatory lifespan. mcp.session_manager.run() must be entered as an async context for the worker's lifetime. Without it, the first request 500s with a non-obvious error. Docs showed it — we copied the pattern.
  • "Session" disambiguation. MCP protocol session ≠ application session. Caught before we accidentally over-coupled stateful mode to "we have multi-step tools later."
  • Stateless + SSE is a real config combo. Initial intuition was "stateless implies no streaming." Docs show stateless_http=True alone keeps SSE; only json_response=True turns SSE off. Useful nuance for explaining the trade space.
